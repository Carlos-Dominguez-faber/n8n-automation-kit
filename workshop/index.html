01 · Conexión
MCP vs API REST
Dos formas de conectar Claude Code con n8n. No son rivales — son complementarias.
🧠
MCP n8n-mcp
Claude "entiende" n8n. Busca nodos, valida configuraciones, crea y edita workflows. Es una capa semántica sobre la API.

buscar nodos validar templates crear/editar
🌐
API REST directa
Claude hace HTTP requests directos. Más rápido, más flexible. Sin overhead del proceso MCP.

monitoreo CI/CD scripts ejecuciones
CapacidadMCPAPI REST
Buscar nodos por keyword✅ search_nodes❌
Validar configuración✅ validate_workflow❌
Crear / editar workflows✅ con validación⚠️ raw JSON
Templates (2700+)✅ search + deploy❌
Ver ejecuciones✅✅
Velocidad100-500ms20-50ms
La regla
MCP cuando construyes y validas. API cuando operas y monitorizas. Los dos viven en el mismo .mcp.json — no son excluyentes.
Configuración mínima
.mcp.json
{
  "mcpServers": {
    "n8n-mcp": {
      "command": "npx",
      "args": ["-y", "n8n-mcp"],
      "env": {
        "N8N_API_URL": "https://TU_N8N/",  // la / al final es obligatoria
        "N8N_API_KEY": "TU_API_KEY",
        "MCP_MODE": "stdio",
        "DISABLE_CONSOLE_OUTPUT": "true"
      }
    }
  }
}

04 · Arquitectura
Sub-workflows: cuándo y cómo
El patrón que vimos en la clase: Router → Sub-workflows por tipo. Determinístico, predecible, debuggeable.
Crear sub-workflow cuando...
✓ La lógica tiene >8 nodos Y es reutilizable
✓ El mismo proceso se invoca desde múltiples workflows
✓ Quieres testear esa parte en aislamiento
✓ Es un tipo de proceso diferente (carousel ≠ reel)
NO crear sub-workflow cuando...
✗ Es una variante menor — usa Switch en su lugar
✗ Solo se usa una vez y tiene <5 nodos
✗ La latencia es crítica (+200ms por executeWorkflow)
Estructura obligatoria de un sub-workflow
1. Execute Workflow Trigger   ← NUNCA un Webhook
2. Validar params del padre   ← IF: tenant_id, content_id, etc.
3. [Lógica específica]        ← lo que hace este sub
4. Set — campos de retorno    ← status, output, error_message
⚠️
El gotcha más común
Al pasar datos con executeWorkflow, type: "string" es OBLIGATORIO en cada campo. Sin él, el campo llega como undefined en el hijo. Este error causa el 80% de los bugs en sub-workflows.
executeWorkflow config
{
  "fields": { "values": [
    { "name": "tenant_id", "stringValue": "={{ $json.tenant_id }}", "type": "string" }
  ]}                              // ↑ type: "string" es obligatorio
}
05 · Microservicios
Cuándo usar microservicios — y cuándo no
La tentación de usar microservicios para todo es real. Aquí está la decisión correcta.
La pregunta que siempre hacerse primero
"¿Puede un Code node de n8n hacer esto en menos de 30 segundos?"
Si sí → Code node. Si no → considera el microservicio.
✅ USA microservicio cuando...
⏱️ Proceso tarda >30s
Video, audio, OCR, ML. El timeout de n8n te va a cortar.
🔧 Runtime específico
Python con sklearn, ffmpeg, Sharp, librerías que no existen en JS.
♻️ Reutilizable entre proyectos
Varios clientes necesitan lo mismo. Un servicio, un mantenimiento.
📈 Escala independiente
Esa pieza recibe 10x más carga que el resto del workflow.
💥 Necesitas aislar el fallo
Si el micro se cae, n8n sigue vivo. En un Code node, no.
❌ NO uses microservicio cuando...
⚡ Un Code node lo hace en <100ms
Transformar datos, calcular scores simples, formatear strings.
🎯 Solo lo usa un workflow
No vale deployar, mantener y monitorear para un solo caso.
🌐 La API ya existe
Hay un servicio externo que hace lo mismo. Usa HTTP Request.
⏰ La latencia importa
Cada microservicio añade 50-200ms de red. Para respuestas síncronas, suma.
🏗️ No tienes infra para mantenerlo
Sin Coolify/Docker, añades complejidad operativa sin beneficio.
Los 3 patrones de comunicación n8n ↔ microservicio
PatrónCuándoOverhead
Síncrono (HTTP Request + espera)Proceso <30s, necesitas el resultado inmediatoBajo
Async + callback (job_id → webhook)Proceso >30s, n8n responde 202 y espera el callbackMedio
Polling (Wait + GET status)El micro no puede hacer callbacks (legacy, APIs externas)Alto
06 · Demo
Demo: Lead Classifier
El workflow que construimos en la clase. Replica los 3 niveles: main workflow → sub-workflow → microservicio.
El caso
Webhook recibe leads {name, email, message}. El sub-workflow llama al microservicio lead-scorer que puntúa el lead por 3 reglas. Score ≥ 50 → calificado.
Arquitectura
Webhook POST /leads/classify
→
IF: name+email+message
→
respondToWebhook 202
executeWorkflow → Lead Scorer Sub
→
IF score ≥ 50
✅ calificado
❌ descartado
Sub-workflow: Execute Workflow Trigger → HTTP Request /score → Set de retorno
El microservicio — 3 reglas de scoring
ReglaPuntosLógica
Email corporativo+40No es gmail/hotmail/yahoo/outlook
Mensaje detallado+30message.length > 50 caracteres
Intención de compra+30Contiene: precio, cotiz, implementar, presupuesto...
Probarlo
bash — lead que SÍ califica
curl -s -X POST https://TU_N8N/webhook/leads/classify \
  -H "Content-Type: application/json" \
  -d '{"name":"Ana","email":"ana@empresa.com","message":"Quiero cotizar la implementación"}' | jq .
bash — lead que NO califica
curl -s -X POST https://TU_N8N/webhook/leads/classify \
  -H "Content-Type: application/json" \
  -d '{"name":"Juan","email":"juan@gmail.com","message":"Hola"}' | jq .
07 · Validación
Validación E2E con Playwright
Validación en dos capas: MCP verifica estructura JSON, Playwright verifica lo que se ve en el canvas.
Setup del perfil Chrome (una vez)
# 1. Crear perfil Chrome dedicado (UI: chrome://settings/ → Perfiles → Agregar)
# 2. Loguearte en n8n y Coolify en ese perfil
# 3. Encontrar el directorio:
ls ~/Library/Application\ Support/Google/Chrome/ | grep -i profile

# 4. En .mcp.json:
"playwright": {
  "command": "npx",
  "args": ["@playwright/mcp@latest", "--browser=chrome",
           "--user-data-dir=/Users/TU_USUARIO/Library/Application Support/Google/Chrome/Default"]
}
Prompt de validación dual
Valida el workflow "Lead Classifier" con perfil runtime.
Luego ábrelo en el browser y toma un screenshot del canvas
para verificar que las sticky notes están visibles y que
el executeWorkflow está conectado al sub-workflow.
🔍
Capa 1 — MCP
validate_workflow detecta nodos mal configurados, campos faltantes, conexiones rotas. Todo en el JSON interno.
👁️
Capa 2 — Playwright
Screenshot del canvas real. Verifica que los sticky notes son visibles, que el toggle está verde, que la UI muestra lo esperado.
