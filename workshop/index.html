01 · Conexión
MCP vs API REST — Dos métodos, dos casos de uso
No es uno u otro — son complementarios. Aprende cuándo usar cada uno.
🧠
MCP n8n-mcp
Claude "entiende" n8n. Sabe qué es un nodo, cómo se conectan, qué significa validar. Actúa como un experto en n8n integrado en Claude.

✓ Validación semántica  ✓ Busca nodos  ✓ 2700+ templates
🌐
API REST directa
Claude hace HTTP requests directos. Más rápido, más flexible. Ideal para leer datos de ejecuciones, disparar workflows desde otro sistema, o como fallback del MCP.

✓ Velocidad  ✓ Sin proceso intermedio  ✓ CI/CD

Buscar nodos por keyword✅ search_nodes❌ No disponible
Crear workflows✅ n8n_create_workflow⚠️ POST raw JSON
Validar configuración✅ validate_node/workflow❌ No disponible
Templates (2700+)✅ search + deploy❌ No disponible
Ver ejecuciones✅ n8n_executions✅ GET /api/v1/executions
Gestionar credenciales❌ No gestiona✅ GET /api/v1/credentials
Velocidad de respuesta~100-500ms (proceso intermedio)~20-50ms (directo)

Las 6 skills de n8n — qué hace cada una
Las skills son conocimiento especializado inyectado en Claude. Las activa automáticamente cuando detecta que las necesita.
📦
¿Cómo se instalan?
Las skills van en ~/.claude/skills/[nombre]/SKILL.md. Claude las carga automáticamente al inicio de cada sesión. No necesitas invocarlas manualmente.
📐
n8n-workflow-patterns
5 patrones arquitecturales probados con sub-archivos por patrón: webhook_processing.md, ai_agent_workflow.md, scheduled_tasks.md, etc.
Invocar cuando: diseñando un workflow nuevo desde cero o eligiendo arquitectura
Arquitectura
🔧
n8n-mcp-tools-expert
Guía de uso de todas las herramientas MCP. Ranking por success rate. Formato correcto de nodeType (nodes-base.slack vs n8n-nodes-base.slack).
Invocar cuando: usando cualquier herramienta MCP de n8n o si Claude no sabe qué tool usar
MCP
⚙️
n8n-node-configuration
Configuración específica por tipo de nodo. Property dependencies (campos que aparecen/desaparecen según otros valores). Evita campos requeridos olvidados.
Invocar cuando: configurando un nodo específico o cuando hay error de campo faltante
Config
🟨
n8n-code-javascript
Sintaxis $input/$json/$node. CRÍTICO: datos webhook bajo $json.body (no $json directo). Formato de retorno: [{json: {...}}]. Built-ins: $helpers.httpRequest(), DateTime.
Invocar cuando: escribiendo código JS en Code nodes
Código
🧮
n8n-expression-syntax
Sintaxis de expresiones {{ }}. REGLA: $('Nodo').item.json.campo en vez de $json (evita roturas al insertar nodos). Fechas con toDateTime().format().
Invocar cuando: escribiendo expresiones o cuando una expresión da undefined
Expresiones
✅
n8n-validation-expert
Interpreta errores de validación. Loop: validate → fix → validate. Profiles: minimal, runtime (recomendado), strict. Detecta falsos positivos.
Invocar cuando: hay errores de validación o antes de deploy a producción

¿Cuándo crear un sub-workflow?
  • ✓
    La lógica tiene más de 8 nodos Y es reutilizable
  • ✓
    El mismo proceso se invoca desde múltiples workflows
  • ✓
    Quieres testear esa parte en aislamiento
  • ✓
    Es un tipo de contenido/proceso diferente (carrusel ≠ reel)
¿Cuándo NO crear uno?
  • ✗
    Es una variante menor — usa Switch en su lugar
  • ✗
    Solo se usa una vez y tiene menos de 5 nodos
  • ✗
    La latencia es crítica (executeWorkflow añade ~200ms)
Estructura obligatoria de un sub-workflow
Todo sub-workflow:

1. Trigger: "Execute Workflow Trigger"
      ↑ NO webhook — este es el entry

2. Validar parámetros recibidos del padre
      ↑ IF: tenant_id != vacío, etc.

3. [Lógica específica del sub]
      ↑ Lo que hace este sub-workflow

4. Set node — campos de retorno al padre
      ↑ status, output_url, credits_used
Pasar datos padre → hijo (multi-tenant)
executeWorkflow node config
// En el Router (padre) — executeWorkflow node:
{
  "workflowId": "={{ $json.sub_workflow_id }}",  // Dinámico desde Supabase
  "mode": "Single Run",
  "fields": {
    "values": [
      { "name": "tenant_id", "stringValue": "={{ $json.tenant_id }}", "type": "string" },
      { "name": "content_id", "stringValue": "={{ $json.content_id }}", "type": "string" },
      { "name": "brand_data", "stringValue": "={{ JSON.stringify($json.brand_data) }}", "type": "string" }
    ]
  }
}

// En el hijo — acceder a los datos:
"={{ $json.tenant_id }}"  // Disponible directamente en $json

// ⚠️ GOTCHA: type: "string" es OBLIGATORIO. Sin él, el campo llega como undefined.
⚠️
Regla multi-tenant — nunca violar
El tenant_id fluye en TODOS los nodos del sub-workflow. Nunca hardcodear credenciales. Siempre: tenant_id → buscar credenciales en tabla tenants → usar credenciales del tenant.

06 · Validación E2E
Chrome profile + Playwright — Debug con sesión activa
El problema: Playwright lanza un Chrome nuevo sin login. La solución: un perfil de Chrome dedicado con las sesiones de n8n y Coolify ya guardadas.
💡
Por qué importa
Sin sesión activa, Playwright ve la pantalla de login — no el workflow. Con un perfil Chrome persistente, entra directamente al canvas del workflow, puede tomar screenshots reales, hacer click en "Execute Workflow" y ver el resultado.
Setup del perfil Chrome (una vez)
bash — setup inicial (solo la primera vez)
# 1. Crear el perfil (en Chrome UI: chrome://settings/ → Perfiles → Agregar)
#    Nombre del perfil: "N8N Coolify Dev"
#    Logearte en pod-1.dopla.app y coolify.dopla.app en ese perfil

# 2. Encontrar el nombre del directorio del perfil
ls ~/Library/Application\ Support/Google/Chrome/ | grep -i profile
# Ejemplo output: Profile 2, Profile 3, Default

# 3. Guardar en .env del proyecto (nunca en git)
echo 'PLAYWRIGHT_CHROME_PROFILE_PATH="/Users/TU_USUARIO/Library/Application Support/Google/Chrome/Profile 2"' >> .env
echo 'PLAYWRIGHT_CDP_PORT=9222' >> .env
Dos modos de uso
Recomendado
Modo A — Perfil persistente
Playwright lanza Chrome con el perfil guardado. Más simple, funciona siempre.
{
  "playwright": {
    "command": "npx",
    "args": [
      "@playwright/mcp@latest",
      "--browser=chrome",
      "--user-data-dir=/Users/TU_USER/Library/Application Support/Google/Chrome/Profile 2"
    ]
  }
}
Alternativo
Modo B — CDP en vivo
Conectar a Chrome ya corriendo. Puedes ver lo mismo que el usuario en tiempo real.
# Lanzar Chrome con debug:
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir="Profile 2"

# En .mcp.json:
"playwright": {
  "args": ["@playwright/mcp@latest",
           "--cdp-endpoint=http://localhost:9222"]
}
